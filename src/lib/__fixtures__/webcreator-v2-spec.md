## 1. Overview

We extend the Webcreator platform with a **lead-generation front of the funnel**: an operator-only flow to discover prospective businesses via Google Places Text Search, qualify them with cheap-then-expensive research tiers, curate them via reject/select, and promote winners into the existing customer + generation pipeline.

**Core user journey**: Operator opens **Potentielle Kunden** → clicks **Neue Suche** → enters country + postal code(s) + vertical/subtype + editable query term → discovery job populates the leads table with discovered businesses (no repo provisioned, no cost beyond Places Text Search) → operator filters/sorts/selects high-potential leads → clicks **Recherche für Auswahl starten** → batch of `refresh` jobs runs extended research (Lighthouse + scrape + RDAP + Wayback + **existing-site screenshot** + **legal-page detection** + **LLM qualitative site assessment**) → operator watches live progress on the research-monitor → clicks **Website generieren / Zu Kunde machen** on a researched lead → project + GitHub repo provisioned → lead becomes a customer ready for the v2 generation pipeline.

**Architecture faithfulness** (✅ VERIFIED against the codebase):
- Reuses the existing `jobs` / `pipeline_runs` / `pipeline_steps` queue, `runWithInstrumentation`, Supabase Realtime, RLS on `customers`, `loggedApiCall` cost tracking, and admin prompts UI verbatim.
- Adds ONE new job type (`discover`) and reuses the existing `refresh` job for extended research.
- Adds ONE new table (`lead_discovery_runs`), one nullable column (`pipeline_runs.batch_id`), and four new columns on `customers` (`lifecycle_stage`, `research_status`, `research_completed_at`, `last_seen_at`).
- Reuses the existing customer detail page for leads (single-table lifecycle).
- AI for judgment only (CLAUDE.md Rule 5): discovery, dedup, legal detection, screenshots are deterministic; only the qualitative site assessment is a model call.

**Technology stack** (✅ VERIFIED):
- **Framework**: Next.js 16 App Router (TypeScript strict)
- **Worker**: Node 22 on Fly.io (`webcreator-worker`)
- **DB**: Supabase Cloud (no local instance — CLAUDE.md)
- **AI**: Anthropic Claude SDK (`@anthropic-ai/sdk`)
- **External APIs**: Google Places v1 REST (`places:searchText`, `places/{id}`), screenshotapi.com, Google PageSpeed, RDAP, Wayback CDX
- **Template recommendation**: None — this is an extension of an existing monorepo, not a scaffolded new app.

## 7. Error Handling & Edge Cases

For every page and interaction, define what happens on failure, with explicit UI surface.

### 7.1 Discovery launcher

- **Network error on POST /api/leads/discover** → Surface: inline (below form) → Copy: "Discovery konnte nicht gestartet werden. Bitte erneut versuchen." → Recovery: button re-enabled, form values preserved
- **Validation error (Zod 400)** → Surface: inline → Copy: "{field}: {message}" → Recovery: user corrects field, error clears on blur
- **403 viewer rejection** → Surface: toast → Copy: "Keine Berechtigung für Discovery." → Recovery: navigate away
- **Subtype not in vertical (422)** → Surface: inline → Copy: "Subtyp passt nicht zur Vertikale." → Recovery: user picks valid subtype

### 7.2 Leads list

- **Initial fetch fails** → Surface: full-page banner → Copy: "Leads konnten nicht geladen werden. {error}" → Recovery: "Erneut versuchen" button reloads
- **Bulk research-batch fails** → Surface: toast → Copy: "Batch konnte nicht gestartet werden. {error}" → Recovery: selection preserved, button re-enabled
- **Bulk reject fails** → Surface: toast → Copy: "Ablehnen fehlgeschlagen." → Recovery: selection preserved
- **Empty list (no leads, no filters)** → Surface: full-page empty state → Copy: "Noch keine Leads entdeckt." → CTA: "Neue Suche starten" opens DiscoveryLauncherDialog
- **Empty (filtered to zero)** → Surface: full-page empty state → Copy: "Keine Leads mit diesen Filtern." → CTA: "Filter zurücksetzen" clears search params

### 7.3 Customer/Lead detail — extended research render

- **`research.legal` undefined** (old v1 research, never re-run) → Surface: inline section → Copy: "Rechtliche Seiten: noch nicht geprüft" → Recovery: button "Recherche aktualisieren" (existing) re-runs research
- **`research.screenshots` undefined** → Surface: inline section → Copy: "Screenshots: nicht verfügbar" → Recovery: same button
- **`research.site_assessment` undefined** → Surface: inline section → Copy: "KI-Bewertung: nicht verfügbar (Recherche aktualisieren)" → Recovery: same button
- **Screenshot image fails to load** → Surface: inline (in section) → Copy: "Bild konnte nicht geladen werden" + placeholder icon → Recovery: none (next research run will retry)

### 7.4 Promotion

- **Promotion API fails** → Surface: toast → Copy: "Promotion fehlgeschlagen: {error}" → Recovery: button re-enabled, lead stays a lead
- **Project insert succeeds but provision-enqueue fails** → Surface: banner on detail page → Copy: "Kunde angelegt, aber Repo-Provisionierung fehlgeschlagen. Im Admin-Bereich nachholen." → Recovery: admin can manually re-enqueue provision job

### 7.5 Research monitor

- **No active batches** → Surface: full-page empty → Copy: "Keine aktiven Recherche-Batches" → CTA: "Zu Potentielle Kunden" → `/potential-customers`
- **Realtime disconnect** → Surface: banner at top → Copy: "Live-Updates getrennt. Aktualisiere automatisch in 8s." → Recovery: automatic reconnect; manual "Jetzt neu laden" button
- **Run with error** → row shows red status icon + tooltip with `pipeline_runs.error` → "Erneut versuchen" link triggers `POST /api/customers/[id]/research` for that lead

### 7.6 Discovery job — worker-side

- **Places API quota exceeded (HTTP 429)** → `loggedApiCall` records the failure, job throws → worker backoff (30s→2m→8m→30m); on final failure `jobs.status='failed'` and `lead_discovery_runs.status='failed'` → next operator-visible discovery launch surfaces this via "Letzte Discovery-Runs" section
- **No results for a postal code** → not an error; logged as info; `total_found` includes 0 for that code
- **All postal codes fail** → run finishes succeeded with `total_found=0`; UI shows "0 Leads gefunden"

### 7.7 Refresh job — extended research per-step failure

✅ VERIFIED existing pattern: failures in individual research sub-calls already degrade gracefully (Lighthouse fails → `research.lighthouse=null`). Apply to new capabilities:
- Legal detection fails (HTML parse error) → `research.legal = { has_impressum: false, has_datenschutz: false }` (signals "not found", per R4)
- Screenshot capture fails → `research.screenshots = { desktop_png_path: null, mobile_png_path: null }` (UI renders "nicht verfügbar")
- Assessment LLM call fails / returns invalid JSON → `research.site_assessment = null`; log error; the refresh job overall still succeeds (the qualitative pass is best-effort, the deterministic data is what gates promotion)

### 7.8 Cross-tab consistency

When the operator opens the leads list in two tabs and bulk-rejects in tab A, tab B will be stale until next refresh. Acceptable for v1; not a regression (no current page is multi-tab-reactive).
