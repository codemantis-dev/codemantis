## 1. Overview

### Purpose
The Knowledge Workbench v1.0 is ~80% shipped but ~20% of its surfaces are dead, misleading, or architecturally orphaned. A user clicking the Readiness tab hits an error state because the backend never shipped. The status-bar Sync button is permanently disabled. The "Undo this sync" button renders but does nothing. The `KbVersionHistoryDrawer` is fully implemented and never mounted. The Ask-KB "Capture as note" button — the compound-loop interaction the strategic spec v2 §3.4 called "critical" — isn't wired anywhere.

This catch-up spec finishes the feature without rebuilding it. It does three things, in order:

**Phase A — Unbreak.** Every dead UI element that a user can click today becomes either functional or removed. No half-buttons, no error tabs, no orphaned drawers. This is the user-visible "stop the bleeding" phase.

**Phase B — Close the compound loop.** Ship the three headline interactions that make the product feel like co-authorship rather than storage: capture-as-note from Ask-KB answers, post-save AI surfacing inside the capture drawer (gradual reveal per spec §4.2), and a KB-influence panel showing "your note shaped BR-049." These are the moments that turn the Workbench from a file system into a thinking partner.

**Phase C — Reconcile.** Six places where the v3 spec and the shipped code disagree. This phase produces decisions — not code — encoded as an appendix delta document. Future specs pull from the reconciled version.

### Core user journey (post-spec)
A PM opens the Workbench. The Readiness tab is gone (hidden honestly, not broken). The status-bar sync button works and opens the same modal as the header. A captured note shows its AI classification + probes + any detected contradictions inline in the capture drawer within 5 seconds of save — no tab switch required. Ask-KB answers carry a "Capture as note" button; clicking it creates a note with references to the cited sources. Clicking a source chip navigates to the source (video at timestamp, note detail, document section, or ticket). Post-sync, the "Undo this sync" button works for 24 hours. A note detail shows "This note contributed to BR-049 in Steuer-News Edition Management." Every surface does what it says it does.

3. Below the body: `"AI thinking…"` skeleton (1-line shimmer, width 80%).
4. T=1-5s: classification pills + any probes + any contradictions stream in via a Realtime subscription on `notes` row + `note_probes` + `note_contradictions` (all tables have realtime enabled per v3 §15.3).
5. A footer appears: `"[Close] [Open in Stream]"`.
6. After 10 seconds of no further updates (and no user interaction), the drawer auto-closes with a toast `"Note saved · [View]"`.
7. User can close explicitly via Esc or the close button at any time.
8. If the user hits Save again while the overlay is showing → discards the current view, opens a fresh empty form for the next note.

## 7. Error Handling & Edge Cases

### Phase A — Unbreak

| Scenario | Behavior | Recovery |
|---|---|---|
| `useRollbackKb` fails (network) | Toast `"Rollback failed: {message}"` | Stay on success screen; button remains clickable for manual retry |
| `useRollbackKb` fails (409 — another sync in progress) | Toast `"Another sync is in progress. Try again in a moment."` | Retry after current sync completes (no auto-poll) |
| Rollback target's `rollback_expires_at < now()` | Button disabled with tooltip "Rollback window expired (24h)." | None — expired is terminal. |
| `useKbVersions` fails in SyncDispatchContainer | Skeleton in place of `appliedVersion` number | Auto-retries via React Query's default retry policy |
| Ask-KB source "Open" clicked with missing navigation identifier (e.g., note source with no `note_id`) | Console warn; fall back to copying the snippet text to clipboard; toast `"Source text copied — could not open directly."` | None — Phase 0 verification should prevent this from shipping |
| User clicks `/workbench/readiness` via direct URL | Redirects to `/workbench/capture` silently | None needed |
| Clarification files referenced in external code (shouldn't happen) | TypeScript compile error | Phase 0 grep verification catches this before deletion |

### Phase B — Compound loop

| Scenario | Behavior | Recovery |
|---|---|---|
| Ask-KB "Capture as note" dialog open, user closes without saving | Confirmation: `"Discard captured answer?"` → yes closes, no stays | N/A |
| Capture-as-note save fails (network / backend) | Inline error below body, body preserved, button re-enabled | Retry by clicking Save again |
| Same answer captured twice (user clicks button twice) | Second click opens dialog pre-filled again; the existing captured chip stays on the message | PM can deliberately capture twice if they want — no dedup |
| Post-save overlay: AI never returns (15s timeout) | Show "AI didn't respond. [Retry analysis] [Close]" | Retry re-queues the `note_proactive_analysis` job; Close dismisses |
