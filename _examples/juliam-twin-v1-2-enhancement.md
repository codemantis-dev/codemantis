```typescript
  callTwinTask: CallTwinTaskFn,
): Promise<ObservationCandidate[]>;
```

**Behavior:**
- Build input: user-identity block + constitution summary + recent activity summary (last 14 days top 20 episodes) + candidate list (kind, rationale, importance, source episode summary, primary_entity name).
- Call `twin.observer_ranking` (Sonnet-class).
- LLM returns: re-ranked candidates with revised `importance` (0-100), revised `rationale` in user voice, and optionally NEW candidates (any kind from `TwinObservationKind` except `mail.needs_reply` — those must come from rule).
- Hard rule (prompt instruction): NEVER emit a candidate referencing the user-self entity (`primary_entity_id === selfEntityId`).
- Fallback: on LLM error, return raw `candidates` unchanged.

**LLM prompt (verbatim):**

```verbatim
You are the triage-ranking module for a personal AI cofounder.

You will receive:
- a USER IDENTITY block (the user's name, aliases, role)
- a brief CONSTITUTION summary (the user's priorities and boundaries)
- a 14-day ACTIVITY SUMMARY
- a list of CANDIDATE observations produced by deterministic rules

Your job:
1. Re-rank importance (0-100) using identity + constitution + recency.
2. Rewrite rationale in second person ("you should …", "this person has been quiet…").
3. SUPPRESS noisy candidates: transactional mail, system mail, mail addressed only to the user themselves, candidates referencing the user-self entity.
4. Optionally EMIT new candidates of kinds:
   commitment.due | decision.needed | cross_topic.pattern |
   project.health_change | constitution.violation
   when the activity summary clearly supports one. Provide rationale and
   source_episode_ids drawn from the supplied summary.

HARD RULE: NEVER emit a candidate whose primary_entity_id equals the
user-self entity id ({selfEntityId}). The user is not their own contact.

Output STRICT JSON only:
{
  "ranked": [
    { "candidate_id": string, "importance": number, "rationale": string }
  ],
  "suppressed": [
    { "candidate_id": string, "reason": string }
  ],
```

**Triggered by:** (system — every observer cron tick, after `buildCandidates`).

## 8. Error Handling & Edge Cases

### 8.A User-self identity errors

- **Error:** `ensureSelfEntity` cannot find or create the self entity (DB error) → **Surface:** banner inside TwinSettings → **Copy:** *"Twin couldn't link your profile to your identity. Try Reconnect MS365 or contact support."* → **Recovery:** Reconnect MS365 button + retry helper.
- **Error:** `resolveRecipients` returns empty after self filter → **Surface:** inline (under triage row Reply modal) → **Copy:** *"This mail has no other recipients — there's nothing to draft a reply to."* → **Recovery:** Dismiss button.

### 8.B Constitution misroute prevention

- **Error:** QA answer arrives with no `target_module_key` and no client-supplied `moduleKey` → **Surface:** none (silent — only the qa_answers row written) → **Copy:** n/a → **Recovery:** n/a. This is intentional per Part 6.B1.a.

### 8.C MS365 connection failures

- **Error:** `invalid_grant` from Microsoft on refresh → **Surface:** banner (TwinConnectionBanner) → **Copy:** *"Microsoft 365 is disconnected — Twin's data is stale since {date}. [Reconnect]"* → **Recovery:** Reconnect button.
- **Error:** rotation upsert failed (DB error) → **Surface:** none initially (background); after 3 consecutive rotation_failed events in 24h, surface in TwinHealth as a red row → **Copy:** *"Token rotation failing — Twin may stop syncing soon. [Reconnect]"* → **Recovery:** Reconnect button.

### 8.D Observer LLM failure

- **Error:** `enrichCandidatesWithLLM` LLM call throws → **Surface:** none (graceful degradation, fall back to raw candidates) → **Copy:** n/a → **Recovery:** next cron tick retries.
- **Error:** LLM returns malformed JSON → **Surface:** TwinHealth → **Copy:** *"Observer ranking failed at {timestamp} — rule-based ranking used"* → **Recovery:** automatic.

### 8.E Routine runner / scheduler health

- **Error:** No scheduler-tick heartbeat for >5 min → **Surface:** TwinHealth → **Copy:** *"Routine scheduler is silent — routines may not run on time"* → **Recovery:** manual function invoke from health page.

### 8.F People / Projects pages

- **Error:** `useTwinPeopleEnriched` query fails → **Surface:** toast → **Copy:** *"Couldn't load people. [Retry]"* → **Recovery:** retry button refetches.
- **Empty (no data):** empty-state card (per §5.G1).
- **Empty (filtered):** filtered-state card with "Clear filters".

### 8.G Memory graph

- **Error:** library load fails → **Surface:** inline panel → **Copy:** *"Couldn't load graph view. [Retry]"* → **Recovery:** retry.
- **Empty (no entities):** *"No entities yet — Twin needs more data."*
- **Performance:** above 1000 nodes, prompt the user to apply filters; canvas-based renderer handles up to ~1k smoothly (decision D3).

### 8.H Project definition

- **Error:** missing required `name` field → **Surface:** inline (field error) → **Copy:** *"Project name is required."* (on submit; clears on input) → **Recovery:** user types.
- **Error:** entity ID resolution fails (team/stakeholder picker references missing entity) → **Surface:** toast → **Copy:** *"One of the selected people is no longer available. Pick again."* → **Recovery:** remove + reselect.

### 8.I Compose self-only edge case (already in 8.A)

### 8.J Sender intent

- **Error:** invalid pattern (no `%` or `@`) → **Surface:** inline (form field) → **Copy:** *"Pattern must look like 'foo@bar.com' or 'noreply@%'"* → **Recovery:** retype.
