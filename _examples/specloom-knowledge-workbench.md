## 1. Overview

### 1.1 What this feature adds

The **Knowledge Workbench** is a new top-level surface in the project workspace that transforms SpecLoom's Knowledge Base from a passive, read-only synthesis artifact into an **active, conversational workspace** for the Product Manager. It is the PM's daily home.

It delivers:

1. **Global capture** — a floating pill-button + Cmd+N shortcut that is always one tap away from anywhere in a project. Opens a drawer with text and voice modes. Persists drafts to IndexedDB so nothing is ever lost.
2. **Proactive AI** — every note save runs a classification + probe generation + contradiction detection pipeline. The UI shows pills, 0–3 follow-up questions, and contradiction banners inline, in-place.
3. **Ask-the-KB** — a streaming RAG chat with source citations. PMs can ask natural-language questions against their project's Knowledge Base and get grounded, cited answers. Also exposes a voice mode for hands-free querying.
4. **Sync-to-KB** — a preview-then-apply flow that shows the exact diff between the current KB and the proposed new version, explained in 4 tabs (Summary, New, Changed, Conflicts). Creates a new `kb_versions` row on apply with a 24-hour rollback window.
5. **Stream view** — a time-ordered stream of all captures with filter chips, search, note-detail drawer, and bulk operations.
6. **Readiness dashboard** — re-uses the existing Completeness Radar and surfaces per-doc-type gaps with "AI Quick Wins" that generate targeted probes.
7. **KB Explorer** — re-uses existing feature-map, data-model, and cross-reference viewers and adds a RulesTable and WorkflowDiagram surface.
8. **Cross-linking** — bidirectional links between notes ↔ KB sections ↔ documents ↔ epic tickets, rendered in a right-side context rail.
9. **CodeMantis Clarification Loop (v2.0)** — an inbox for clarifications generated by CodeMantis during implementation. PMs answer in the Workbench; answers flow back as KB updates.
10. **Polish (v2.1)** — entity graph visualization, KB diff viewer, note templates, email intake.

### 1.2 Why it matters

SpecLoom's current flow is **batch-oriented**: PM uploads inputs → worker synthesizes → PM reviews output. The Workbench flips this into a **continuous-intake** model:

- PMs capture the messy reality of their week (1:1 notes, customer calls, whiteboard photos) instantly.
- AI enriches every capture at save time (classification, probes, contradiction detection).
- Sync-to-KB is a human-in-the-loop gate, not a silent overwrite.
- Asking the KB becomes a daily habit, not a one-off artifact view.

This feature set is the **most significant UX evolution** in SpecLoom's roadmap. The spec's own framing: *"the PM's cognitive prosthesis, not a CRUD screen."*

## 3. Data Model Changes

All SQL below is additive. No table drops, no column drops, no type changes on existing columns. All new tables have RLS enabled and scoped by `project_id`.

### 3.1 `notes` — primary capture table

```sql
CREATE TABLE notes (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id  uuid NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  created_by  uuid NOT NULL REFERENCES auth.users(id),
  body        text NOT NULL,                          -- raw or voice-cleaned markdown
  raw_voice_transcript text,                          -- pre-cleanup whisper output (voice only)
  source_type text NOT NULL CHECK (source_type IN ('text','voice','email','paste','imported')),
  tags        text[] NOT NULL DEFAULT '{}',
  status      text NOT NULL DEFAULT 'captured'
              CHECK (status IN ('captured','analyzed','synced','archived','deleted')),
  synced_to_kb_version uuid REFERENCES kb_versions(id),
  deleted_at  timestamptz,
  created_at  timestamptz NOT NULL DEFAULT now(),
  updated_at  timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX idx_notes_project_created ON notes(project_id, created_at DESC)
  WHERE deleted_at IS NULL;
CREATE INDEX idx_notes_status ON notes(project_id, status);
ALTER TABLE notes ENABLE ROW LEVEL SECURITY;
CREATE POLICY notes_owner_rw ON notes
  USING (project_id IN (SELECT id FROM projects WHERE owner_id = auth.uid()))
  WITH CHECK (project_id IN (SELECT id FROM projects WHERE owner_id = auth.uid()));
```

## 6. API / Data Layer Changes

### 6.1 Frontend API module conventions

All new API modules follow the existing pattern in `src/lib/api/knowledge-base.ts` ✅ VERIFIED:
- Typed functions, throw on error, no wrapping abstraction.
- Types imported from `src/lib/database.types.ts` (after running `pnpm supabase gen types`).

### 6.2 New endpoints

#### Notes

| Endpoint | Method | Function | Purpose |
|---|---|---|---|
| `/notes-create` | POST | `createNote(input)` | Insert note + dispatch `note_proactive_analysis` job |
| `/notes-update` | PATCH | `updateNote(id, patch)` | Update body/tags/status |
| `/notes-delete` | POST | `deleteNote(id)` | Soft-delete |
| `/notes-list` | GET | `listNotes(projectId, filters)` | Cursor-paginated list |
| `/notes-voice-upload` | POST | `uploadVoiceNote(blob)` | Upload audio → dispatch `note_voice_transcription` |

All notes endpoints:
- Require `Authorization: Bearer <user JWT>` and `X-Idempotency-Key` on mutations.
- Enforce RLS via `project_id`.
- Slow-response (>3s): the frontend shows a persistent loading skeleton + "Saving…" toast until response or timeout (30s).

#### Probes & contradictions

| Endpoint | Method | Function |
|---|---|---|
| `/note-probes-answer` | POST | `answerProbe(probeId, answer)` |
| `/note-probes-dismiss` | POST | `dismissProbe(probeId)` |
| `/note-contradictions-resolve` | POST | `resolveContradiction(id, resolution)` |

#### Ask-KB

| Endpoint | Method | Function |
|---|---|---|
| `/ask-kb` | POST (SSE) | `askKbStream(input)` → yields `{ delta, source_chunk?, done }` |
| `/ask-kb-sessions` | GET/POST/PATCH/DELETE | CRUD on `conversations` rows with `conversation_type='ask_kb'` |

**Streaming contract:** identical to `qa-conversation` ✅ VERIFIED — server-sent events with JSON lines. Final event contains `<ask_kb_metadata>` JSON block with the sources list.

#### Sync

| Endpoint | Method | Function |
|---|---|---|
| `/notes-sync-preview` | POST | `previewSync(projectId)` → returns `{ sync_job_id }` |
| `/notes-sync-jobs/:id` | GET | `getSyncJob(id)` → status + `preview_payload` |
| `/notes-sync-apply` | POST | `applySync(projectId, { resolved_conflicts })` |
| `/kb-versions` | GET | `listKbVersions(projectId)` |
| `/kb-versions-rollback` | POST | `rollbackKb(versionId)` |

**Preview caching:** the preview payload is cached for 60 minutes on the `sync_jobs` row. If a preview is requested within that window, return the cached payload.

**Concurrency:** the partial unique index on `sync_jobs` (§3.7) prevents two simultaneous syncs per project. API returns `409 Conflict` with `{ reason: "another_sync_in_progress", existing_job_id }` if one is active.

### 6.3 Job types (worker)

5 new job types in `processing_jobs.job_type`:

1. `note_proactive_analysis` — runs classify → probe → contradict for one note
2. `note_voice_transcription` — Whisper + Gemini cleanup
3. `ask_kb_rag` — retrieval + generation (streamed through Edge Function)
4. `notes_sync_preview` — diff engine only, writes `preview_payload`
5. `notes_sync_apply` — full KB rebuild + `kb_versions` insert

### 6.4 Modified: `job-orchestrator`

Extend the handler switch (currently handles video/document/synthesis jobs) with 5 new branches. On `notes_sync_apply` completion, the orchestrator does **not** write the `kb_versions` row itself — the worker does, inside the same transaction as the KB update.

### 6.5 RLS policies

All new tables enable RLS. All policies scope by `project_id IN (SELECT id FROM projects WHERE owner_id = auth.uid())`. Where the table has no direct `project_id` (e.g. `note_embeddings`, `note_classifications`, `note_probes`, `note_contradictions`), policies route through `notes`.

### 6.6 Slow-response behavior (>3s)

- Notes create: optimistic insert into query cache → server eventually confirms. Post-save AI pills/probes/contradictions stream in via Realtime subscription, not part of the save response.
- Sync preview: returns `202 Accepted` with `sync_job_id`; frontend subscribes to `sync_jobs` Realtime.
- Ask-KB: SSE stream — partial tokens render as they arrive; no timeout.

## 7. Integration Points

### 7.1 Cross-component data flow

**Capture → Post-save AI:**
```
CaptureTextMode submits → POST /notes-create → note row + job queued
  → Optimistic insert into useNotes cache
  → job-orchestrator runs note_proactive_analysis
  → Worker writes note_classifications, note_probes, note_contradictions
  → Realtime publishes to client
  → ProbeStack, ClassificationPills, ContradictionBanner re-render
```

**Sync flow:**
```
SyncCtaButton click → SyncPreviewModal opens
  → POST /notes-sync-preview → sync_jobs row created (phase=preview)
  → useNotesSync subscribes to sync_jobs → status becomes 'running'
  → worker writes preview_payload → status='completed'
  → SyncPreviewModal fetches preview_payload → renders 4 tabs
  → user resolves conflicts → Apply click
  → POST /notes-sync-apply with resolved_conflicts
  → sync_jobs row created (phase=apply)
  → worker rebuilds KB + inserts kb_versions row → updates sync_jobs
  → SyncApplySuccess shown → Status bar shows new version
```

**Ask-KB flow:**
```
AskKbComposer send → POST /ask-kb (SSE)
  → Edge Function dispatches ask_kb_rag job to worker
  → Worker retrieves top-k from knowledge_base_entries + note_embeddings
  → Assembles prompt → streams tokens back through Edge Function
  → AskKbConversation renders delta events in real time
  → Final event contains sources → AskKbSourcesPanel renders cited chunks
```

### 7.2 Cross-link integration (v1.4)

Bidirectional link registry: a new table `cross_links` is NOT required — links are derived from:
- `notes.synced_to_kb_version` → KB sections
- `note_contradictions.conflicts_with_type + conflicts_with_id`
- Vector-similarity lookups between `note_embeddings` and `knowledge_base_entries.embedding` / `documents.content_embedding`

`WorkbenchContextRail` performs these lookups on demand (cached per-note in React Query).

### 7.3 Completeness Radar reuse (v1.2)

## 8. Error Handling & Edge Cases

### 8.1 Capture

| Scenario | Behavior |
|---|---|
| Network drops during save | Draft stays in IndexedDB; inline error; retry button; save button re-enabled |
| Empty body on Save | Save button disabled; inline hint: "Add a note before saving" |
| Voice transcription fails | Banner in drawer: "Voice transcription failed — try again or type your note"; audio blob preserved for retry |
| Save succeeds but proactive analysis fails | Note appears in Recent Captures with status=captured (not analyzed); silent retry 3x then surface a toast "Analysis temporarily unavailable" |
| Duplicate Cmd+N press while drawer is open | Ignored (drawer already open) |
| IndexedDB unavailable (Safari private mode) | Fall back to in-memory draft; warn once with a toast |

### 8.2 Ask-KB

| Scenario | Behavior |
|---|---|
| Stream interrupted | Partial answer preserved; `[Regenerate]` button appears |
| No sources found | AI answers "I don't have enough information yet" + suggestion to capture more notes |
| Rate limit hit | Banner: "Too many questions — try again in Ns" |
| Empty input | Send disabled |

### 8.3 Sync

| Scenario | Behavior |
|---|---|
| Another sync is already running | 409 Conflict → modal shows "Another sync is in progress — wait for it to finish" with link to active job |
| Preview times out (>120s) | Banner: "Preview took too long — try again"; `[Retry]` |
| Conflicts unresolved | Apply button disabled; Conflicts tab shows red-dot badge |
| Apply fails mid-transaction | Worker rolls back; `sync_jobs.status='failed'` + `error_message` populated; UI shows: "Sync failed — {error_message}. No changes applied."; `[Retry]` |
| Rollback after 24h window | Button disabled; tooltip: "Rollback window expired" |
| Rollback during a live Ask-KB stream | Stream completes with current sources; next query uses new version |

### 8.4 Probes & contradictions

| Scenario | Behavior |
|---|---|
| User tries to sync a note with a `block`-severity unresolved contradiction | Sync is blocked; Conflicts tab highlights the note; inline CTA: "Resolve contradiction before syncing" |
| Probe expires unanswered | Moves to `status='expired'`, greyed out, user can still open and answer retroactively |
| All probes dismissed | No probe stack rendered |

### 8.5 Global error UI conventions

- Toasts: transient (3s), top-right, via `sonner`
- Banners: inline at the top of the affected surface, persistent
- Dialog errors: red-bg strip at top of modal body, submit button re-enabled, fields preserved
- Every recovery path uses an existing shadcn `Button` variant — no new button variants.
