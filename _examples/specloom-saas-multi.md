## 1. Overview

This feature completes SpecLoom's multi-tenancy infrastructure and introduces a SaaS Admin layer. The application already has skeleton multi-tenancy (`tenants` table, `tenant_id` on key tables, `user_role` enum), but no tenant provisioning, no invitation system, and no admin surface. This spec ships the full end-to-end SaaS stack:

**What is added:**
1. **Tenant auto-provisioning** — Public signup creates a trial tenant; a `/create-workspace` step names the workspace.
2. **Invitation system** — Tenant owners/admins invite users by email via OTP (no magic links). `/accept-invite` page. First-time users set their own password.
3. **SaaS Admin layer** — `/admin/*` routes visible only to users with `saas_admin` role. Manage all tenants, users, platform settings, and full monitoring across all tenants.
4. **Scoped monitoring** — Non-admin users see only their tenant's jobs in Live Pipeline and Job Explorer, and their costs in Cost Analytics. Log Viewer and System Health are hidden.
5. **Email sender update** — All transactional emails change from `noreply@soc-plus.com` to `info@soc-plus.com`.

**What is NOT changed:** The AI pipeline, document generation, knowledge base, synthesis, or any project-level features. This is a pure auth/tenancy/admin layer.

### 3.3 New `tenant_invitations` table

```sql
CREATE TABLE IF NOT EXISTS "public"."tenant_invitations" (
  "id"          UUID DEFAULT gen_random_uuid() NOT NULL,
  "tenant_id"   UUID NOT NULL REFERENCES "public"."tenants"("id") ON DELETE CASCADE,
  "email"       TEXT NOT NULL,
  "role"        "public"."user_role" NOT NULL DEFAULT 'member',
  "invited_by"  UUID NOT NULL REFERENCES "public"."users"("id") ON DELETE CASCADE,
  "status"      TEXT NOT NULL DEFAULT 'pending'
    CONSTRAINT "tenant_invitations_status_check"
      CHECK (status IN ('pending', 'accepted', 'cancelled', 'expired')),
  "created_at"  TIMESTAMPTZ NOT NULL DEFAULT now(),
  "expires_at"  TIMESTAMPTZ NOT NULL DEFAULT (now() + INTERVAL '7 days'),

  CONSTRAINT "tenant_invitations_pkey" PRIMARY KEY ("id"),
  CONSTRAINT "tenant_invitations_tenant_email_pending_uniq"
    UNIQUE NULLS NOT DISTINCT ("tenant_id", "email") -- only one active invite per email per tenant
);

CREATE INDEX IF NOT EXISTS "idx_tenant_invitations_tenant_id"
  ON "public"."tenant_invitations" ("tenant_id");

CREATE INDEX IF NOT EXISTS "idx_tenant_invitations_email"
  ON "public"."tenant_invitations" ("email");

ALTER TABLE "public"."tenant_invitations" ENABLE ROW LEVEL SECURITY;

-- Tenant owners and admins can view their own tenant's invitations
CREATE POLICY "tenant_members_view_invitations" ON "public"."tenant_invitations"
  FOR SELECT USING (tenant_id = public.get_user_tenant_id());

-- saas_admin sees all
CREATE POLICY "saas_admin_all_invitations" ON "public"."tenant_invitations"
  FOR ALL USING (public.is_saas_admin());

GRANT SELECT, INSERT, UPDATE ON "public"."tenant_invitations" TO "authenticated";
GRANT ALL ON "public"."tenant_invitations" TO "service_role";
```

### 3.6 New `is_saas_admin()` helper function

Used in all RLS policies throughout the database:

```sql
CREATE OR REPLACE FUNCTION "public"."is_saas_admin"()
RETURNS BOOLEAN
LANGUAGE "sql" STABLE SECURITY DEFINER
SET "search_path" TO 'public'
AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.users
    WHERE id = auth.uid() AND role = 'saas_admin'
  );
$$;
```

## 6. API / Data Layer Changes

### 6.1 New Edge Function: `tenant-onboard`

**Route:** `POST /functions/v1/tenant-onboard`  
**Auth:** Requires valid user JWT (ES256)  
**Purpose:** Creates a tenant and assigns the calling user as owner.

**Request body:**
```typescript
{ name: string }  // 2–80 chars
```

**Logic:**
1. Verify JWT → get `user.id`
2. Check `public.users.tenant_id` for this user — if already set, return 409 Conflict
3. Validate `name` (2–80 chars, not blank)
4. Generate slug: `slugify(name) + '-' + randomHex(6)` (unique, retry on collision)
5. Read `saas_platform_settings` for `default_plan` and `trial_duration_days`
6. INSERT into `tenants`: `{ name, slug, plan: default_plan, status: 'trial', trial_ends_at: now() + trial_duration_days }`
7. UPDATE `public.users` SET `tenant_id = new_id, role = 'owner'` WHERE `id = user.id`
8. Return `{ tenant: { id, name, slug, plan, status, trial_ends_at } }`

**Slow response (>3s):** Frontend loading spinner stays active. No timeout on client; server times out at 25s (Supabase default).

**Deploy:** `supabase functions deploy tenant-onboard --no-verify-jwt`

### 6.2 New Edge Function: `send-invitation`

**Route:** `POST /functions/v1/send-invitation`  
**Auth:** Requires valid user JWT. Caller must be `owner` or `admin` in their tenant.

**Request body:**
```typescript
{ email: string, role: 'admin' | 'member' | 'viewer' }
```

**Logic:**
1. Verify JWT → get caller's `user.id`, `tenant_id`, `role`
2. Assert caller is `owner` or `admin` — else 403
3. Check invitee is not already a member of this tenant
4. Upsert into `tenant_invitations` — if a pending invite exists for this email + tenant, resend (update `created_at` and `expires_at`, set status back to `pending`)
5. Generate 6-digit OTP, store in `otp_codes` with `type = 'team_invitation'`, `expires_at = now() + 48h`
6. Send invitation email via Resend (template below — Section 6.6)
7. Return `{ invitation: { id, email, role, expires_at } }`

**Rate limit:** 10 invitations per tenant per hour (checked against `otp_codes` for `team_invitation` type).

### 6.3 New Edge Function: `accept-invitation`

**Route:** `GET /functions/v1/accept-invitation?invite={inviteId}` — returns invite metadata (public, no auth)  
**Route:** `POST /functions/v1/accept-invitation` — accepts the invite

**GET request:**
- Reads `tenant_invitations` by `id`
- Returns `{ email, tenantName, inviterName, role, status, expires_at }` (stripped — no sensitive data)
- If not found or status ≠ `pending` or past `expires_at` → returns `{ error: 'invalid' | 'expired' | 'accepted' }`

**POST request body:**
```typescript
{ inviteId: string, otp: string, password: string }
```

**Logic:**
1. Load `tenant_invitations` record by `inviteId` — validate pending + not expired
2. Verify OTP via shared verify-otp logic (email + code + type = `team_invitation`)
3. Check if a Supabase Auth user exists for this email:
   - If NO: `supabase.auth.admin.createUser({ email, password, email_confirm: true })`
   - If YES: update password via `supabase.auth.admin.updateUserById()`
4. Upsert into `public.users`: `{ id: newUserId, email, tenant_id, role }` (trigger may have already created the row — upsert handles both cases)
5. Mark `tenant_invitations.status = 'accepted'`
6. Mark `otp_codes.verified = true`
7. Return `{ accessToken, refreshToken }` so the client can auto-sign-in

**Deploy:** `supabase functions deploy accept-invitation --no-verify-jwt`

## 7. Integration Points

### 7.1 Signup → workspace creation flow

```
User fills signup form (name, email, password, optional company_name)
→ auth.signUp() fires
→ handle_new_user trigger creates public.users (tenant_id = null)
→ preferences.company_name = company_name stored via update after signup
→ User redirected to /verify-email
→ User enters OTP → verify-otp EF confirms email → sets preferences.email_verified = true
→ Auth state change fires → fetchProfile() → profile.tenant_id === null
→ ProtectedRoute detects: emailVerified + !hasTenant + !isSaasAdmin → redirect /create-workspace
→ User names workspace → tenant-onboard EF creates tenant + sets tenant_id + role=owner
→ Navigate to /onboarding
```

### 7.2 Invitation → new user onboarding flow

```
Tenant owner/admin opens Settings → Team
→ Enters email + role → [Invite]
→ send-invitation EF: creates tenant_invitations row + OTP in otp_codes + sends email
→ Pending invite appears in Team Settings list

Invitee opens email → clicks /accept-invite?invite={id} URL
→ AcceptInvitePage loads: GET accept-invitation → shows tenant name, inviter, role, email
→ User enters OTP + new password → [Accept invitation]
→ POST accept-invitation: verify OTP → create Supabase auth user → create public.users record
  with tenant_id + role → mark invitation accepted → return tokens
→ Client calls supabase.setSession(tokens) → auth state change → navigate to /onboarding
```

### 7.3 SaaS Admin data flow

```
Admin opens /admin/tenants
→ useAdminTenants() calls getAdminTenants()
→ SELECT on tenants — RLS: is_saas_admin() = true → returns all rows
→ Table renders with plan/status badges

Admin clicks [Suspend] on a tenant row
→ updateTenantStatus(tenantId, 'suspended') → UPDATE tenants SET status='suspended'
→ React Query invalidates ['admin-tenants']
→ Row updates with 'Suspended' badge
→ toast.success("Tenant suspended")
```

### 7.4 Monitoring scoping cross-component flow

```
Non-admin user opens /monitoring/pipeline
→ usePipelineJobs() calls useAuth() → isSaasAdmin = false → tenantFilter = profile.tenant_id
→ Supabase query: processing_jobs.select('*').eq('tenant_id', profile.tenant_id)
→ RLS double-checks: tenant_id = get_user_tenant_id() → passes
→ Only this tenant's jobs render

saas_admin opens /admin/monitoring + selects tenant "Acme Corp"
→ usePipelineJobs({ tenantIdFilter: 'acme-uuid' })
→ query: .eq('tenant_id', 'acme-uuid')
→ RLS: is_saas_admin() = true → passes
→ Only Acme Corp's jobs render
```

## 8. Error Handling & Edge Cases

### 8.1 Invitation edge cases

| Scenario | Behavior |
|---|---|
| Email already a member of this tenant | `send-invitation` returns 409 — "This email is already a team member" |
| Invitation expired (past 7 days) | `accept-invitation` GET returns `{ error: 'expired' }` — AcceptInvitePage shows "This invitation has expired. Ask {tenant} to send a new one." |
| Invitation already accepted | GET returns `{ error: 'accepted' }` — shows "This invitation has already been used. Sign in instead." |
| Invitee already has a Supabase Auth account | `accept-invitation` updates their password and reassigns tenant (they may leave their old tenant — this is expected; handle_new_user trigger won't fire again) |
| OTP wrong code | Stay on AcceptInvitePage, show "Invalid code. N attempts remaining.", re-enable form |
| OTP max attempts (5) | OTP deleted — show "Too many attempts. Ask your team admin to resend the invitation." [Contact admin button] |
| Tenant at seat limit | Not enforced in this phase (tiers are labels only) |

### 8.2 Workspace creation edge cases

| Scenario | Behavior |
|---|---|
| `tenant-onboard` called when tenant already exists | Returns 409 — ProtectedRoute will re-fetch profile and see `tenant_id` is set — stops redirect loop |
| OAuth user logs in (no email verification step) | ProtectedRoute detects `!hasTenant → /create-workspace` |
| Name collision on slug | `tenant-onboard` retries with a different random suffix (max 5 retries, then error) |
| Network error on workspace creation | Inline error "Failed to create workspace. Try again." — button re-enabled |

### 8.3 Admin operation errors

All admin mutations show `toast.error("Operation failed. Try again.")` on error and do NOT optimistically update. Confirmation dialogs (suspend, delete) re-enable their buttons on error — dialog stays open.

### 8.4 Email sender update — recovery

If Resend rejects the new `from:` address `info@soc-plus.com`, the Edge Function returns 503 "Email service not configured" (same as existing behavior). The Resend domain must have `info@soc-plus.com` verified.
