# Codex app-server protocol — empirical schemas

These JSON Schemas are the **authoritative wire-format reference** for
the `codex app-server` JSON-RPC protocol. They are generated by the
Codex binary itself, not transcribed from docs:

```
mkdir -p docs/internal/codex-app-server-schemas
codex app-server generate-json-schema --out docs/internal/codex-app-server-schemas
```

Re-run that command after every Codex CLI bump and commit the diff. Any
divergence between these files and the matching production code paths
(`src-tauri/src/agents/codex/*.rs`) is a wire-format drift — trust the
schema, update the code.

This is the Codex analog of `docs/internal/cli-2.1.126-protocol-report.md`
for Claude (which carries per-scenario NDJSON evidence from the
`cli_protocol_capture` harness). Same pattern, different agent.

## Files of interest

| Schema | Source of truth for |
|---|---|
| `v2/ThreadStartParams.json` | `thread/start` params — `sandbox`, `approvalPolicy`, `personality`, `model`, etc. **Confirmed kebab-case** for `SandboxMode` / `AskForApproval` enums (the Phase 2 spec was wrong about camelCase). |
| `v2/ThreadResumeParams.json` | `thread/resume` params — same shape as ThreadStart plus a required `threadId`. |
| `v2/TurnStartParams.json` | `turn/start` params — `input` (array of `UserInput`), `effort` (`none`/`minimal`/`low`/`medium`/`high`/`xhigh`), `model`, plus per-turn overrides. |
| `v2/TurnInterruptParams.json` | `turn/interrupt` params. |
| `v1/InitializeParams.json` | `initialize` params — `clientInfo` (with `name` + `version` required, `title` optional) + `capabilities`. |
| `codex_app_server_protocol.v2.schemas.json` | Bundled v2 protocol document. |

## How to verify a wire-shape assumption

1. Run `codex app-server generate-json-schema --out /tmp/codex_schema`
2. Find the relevant `*Params.json` or `*Notification.json` file
3. Compare against the matching Rust call in `src-tauri/src/agents/codex/`
4. If they disagree, the schema wins — update the Rust.

Examples of empirically-derived findings that contradicted the Phase 2
spec document:

- `SandboxMode` is **kebab-case** (`read-only`/`workspace-write`/
  `danger-full-access`), not camelCase. The spec said `readOnly` etc.
  → `CodexSandbox::as_codex_wire()` in `agents/mod.rs`.
- `AskForApproval` is **kebab-case** (`untrusted`/`on-failure`/
  `on-request`/`never`), not `onRequest`. Spec was wrong.
  → `CodexApproval::as_codex_wire()`.
- `personality` and `serviceName` **are** valid `ThreadStartParams`
  fields. v1.3.1 #5 wrongly suspected them; the empirical schema
  confirmed they're accepted (and `personality` enum is
  `none | friendly | pragmatic`).
- `cwd` is **optional** on `ThreadStartParams` (no required fields).
  Only `ThreadResumeParams` has `threadId` as required.
- `ReasoningEffort` enum is `none | minimal | low | medium | high |
  xhigh`.
